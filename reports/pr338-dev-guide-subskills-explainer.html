PR #338 Explainer：把 dev-guide 经验沉淀成 4 个 nested reference

本 PR 从一个 release-workflow subskill 扩展成一个更大的 lingtai-dev-guide 文档 PR：在已有 release-workflow reference 之外，再新增 3 个 nested reference，把 skill-audit 收敛出的高频开发者/运维经验固化进现有 skill 路由系统，并保持根 router 简短。

结论（conclusion-first）：这是纯文档/技能路由改动，不改运行时代码逻辑。新增 4 个 reference/<topic>/SKILL.md 子技能（release-workflow 已在前序提交，本轮再加 3 个），更新根 router 的 catalog / routing table / layout / version，并同步更新 nested-reference 测试与本地 HTML explainer。所有 shipped 文档已去私有化（无 secrets、无私有绝对路径，仅保留占位符和 repo-relative 示例）。

新增的三个 nested reference

1. runtime-self-check

reference/runtime-self-check/SKILL.md

  • 合并：agent-runtime-self-check / agent-runtime-import-probe / lingtai-active-tui-rebuild + 运行环境/源检查、active binary/dev-mode symlink、clean release worktree 重建、MCP/addon 来源与 tool surface、post-refresh 诊断、安全证据回报。
  • 读-only 诊断；唯一允许的写操作是被授权的开发者重建（make build / editable 重装）。
  • secrets 一律 <REDACTED>，私有路径泛化为 <your-lingtai-checkout> / ~/.lingtai-tui/...。

2. pr-review-deliverables

reference/pr-review-deliverables/SKILL.md

  • 合并：lingtai-pr-review-gate / github-pr-with-html-explainer / html-deliverable-template + 相关 release-style-audit 交付要素。
  • 覆盖 PR readiness、多模型/daemon/Claude read-only review gate、自包含本地 HTML explainer、PR body hygiene（含 gh pr edit GraphQL 警告 note）、source-labeled deliverable + validation/syntax check、maintainer 授权边界。
  • 不复刻 release-workflow：release-specific 发布交给 reference/release-workflow/SKILL.md。

3. skill-stewardship

reference/skill-stewardship/SKILL.md

  • 合并：skill-benchmark-framework / 归一 skill 蒸馏元方法 / skill 审计 triage / session-journal 作为 skill 种子 / shared-library 候选 grooming。
  • 覆盖何时写 skill、router vs nested-reference 模式、去私有化/参数化、投产前轻量 benchmark/checklist、PR-ready skill cleanup。
  • 交叉链接 skills-manual 处理通用 skill 编写，不复刻整本 manual。

为什么这样做

  • skill audit（daemon A/B/C + skill-audit-data.json）显示 runtime self-check、PR/HTML 交付、skill 蒸馏/质量门禁是全网最高频、收敛性最强的开发者经验（runtime self-check 被 8–11+ agent 独立提出）。把它们沉淀为 lingtai-dev-guide 的 nested reference，可随 TUI utilities 一起分发，而不是停留在各 agent 的临时 custom skill。
  • 保持 progressive disclosure：根 router 仍短（catalog + routing table + 交叉链接），细节落在 reference/<topic>/。
  • 避免重复：三项都交叉链接已有 reference（release-workflow / security-audit / debug-troubleshoot / contributing / skills-manual），不复刻其内容。
  • 去私有化是硬要求：shipped 文档不含具体人名、dev-1/dev-2/dev-3 私有路径、secrets、Telegram/email；仅用占位符与 repo-relative 示例。
