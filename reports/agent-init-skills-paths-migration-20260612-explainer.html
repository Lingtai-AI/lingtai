PR #312 follow-up · project migration m038

Repair skills.paths in already-materialized agent init.json

PR #312 fixed the preset editor bug that deleted skills.paths on model switch, and m037 repaired the saved presets it damaged. Jason's review pointed out the target that actually matters: agents materialized while the bug was live carry the same damage in their own init.json, and they keep running with it regardless of the repaired preset. m038 is the per-project companion that patches those files directly.

What m038 does

On the project's next TUI launch, the migration walks every agent directory under .lingtai/ (skipping hidden dirs and human/) and silently ensures manifest.capabilities.skills.paths exists in each init.json, restoring the defaults when missing:

"skills": {
  "paths": ["../.library_shared", "~/.lingtai-tui/utilities"]
}

Same rules as m037, applied per agent via the shared patchPresetSkillsPathsMap helper:

  • Existing skills.paths values (custom overrides) are preserved exactly — the file isn't even rewritten.
  • Other skills config keys and all sibling capability entries (web_search, bash, …) are untouched.
  • A missing manifest.capabilities map is created; a non-object capabilities (or manifest) means that file is skipped safely.
  • Malformed init.json is skipped with a stderr note only; one broken agent never stalls the migration.
  • Writes are atomic (temp file + rename), matching every other migration in the package.

Migration etiquette

  • Both registries bump CurrentVersion 37 → 38 with m038 agent-init-skills-paths.
  • The migration touches shared on-disk state (agent init.json) and both binaries run migrations against the shared meta.json version space, so the repair is implemented with identical logic in both tui/internal/migrate and portal/internal/migrate — whichever binary opens the project first performs the one-time repair. A no-op portal stub would have let a portal-first launch stamp the project at version 38 and silently skip the repair forever.

Tests

  • TestMigrateAgentInitSkillsPathsPatchesMultipleAgents — two agents (paths lost vs. whole skills entry lost) both repaired; sibling web_search config survives.
  • TestMigrateAgentInitSkillsPathsCreatesMissingCapabilitiesMap — an init.json with no capabilities map at all gets one with the default skills paths.
  • TestMigrateAgentInitSkillsPathsPreservesCustomPaths — custom skills.paths leaves the file byte-identical.
  • TestMigrateAgentInitSkillsPathsSkipsMalformedAndNonMapCases — truncated JSON, list-shaped capabilities, and string manifest are all left unchanged while a valid sibling agent is still patched.
  • Portal: TestRunFrom37PatchesAgentInitAndAdvancesTo38 pins the portal-opened-first scenario — Run on a version-37 project repairs agent init.json and advances meta.json to 38 — plus portal copies of the preserve-custom-paths and malformed/non-map skip tests.
