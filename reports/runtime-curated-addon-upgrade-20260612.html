Runtime upgrade now refreshes LingTai curated MCP packages

Repo: Lingtai-AI/lingtaiBranch: fix/runtime-eager-addon-upgradeDate: 2026-06-12

结论

这个改动修的是 Feishu stale PyPI 包暴露出来的长期问题：pip install --upgrade lingtai 默认只升级指定包，依赖如果仍满足 loose lower bound 就不会动。因此 runtime 以后不再只升级 lingtai，而是显式把 LingTai curated MCP 包一起列为升级目标。

lingtai
lingtai-imap
lingtai-telegram
lingtai-feishu
lingtai-wechat
lingtai-whatsapp

范围是刻意收窄的：不使用全局 eager dependency upgrade，不会让任意三方传递依赖都跟着升级。

行为变化

普通 PyPI runtime

  • 启动/doctor 检查 lingtai 版本。
  • 同时检查五个 curated MCP 包的 installed/latest 版本。
  • 如果 lingtai 或任一 curated MCP 过旧，运行 scoped upgrade。

editable/dev runtime

  • 继续保护 editable lingtai，不 clobber 本地 kernel checkout。
  • 但仍检查并升级 curated MCP wheel 包。
  • force/doctor 模式也只修 addon，不把 lingtai 换回 wheel。

实现点

  • tui/internal/config/venv.go 新增 curated MCP 包列表和版本检查。
  • runtimeUpgradeCommand：升级命令变成 lingtai + curated MCP packages。
  • curatedMCPUpgradeCommand：editable runtime 下只升级 curated MCP 包，不含 lingtai。
  • EnsureVenv / dev editable install 初次安装也显式安装 curated MCP 包。
  • 测试 fake runner 增加 importlib.metadata.version() 与 per-package PyPI JSON 支持。
  • ANATOMY.md 更新，明确不是 arbitrary dependency eager upgrade。

验证

已通过 focused 测试：

cd tui && go test ./internal/config

提交/PR 前还会继续跑：

git diff --check
cd tui && go test ./...

边界

  • 这个 PR 解决“runtime upgrade 时 addon 不跟新”的机制问题。
  • Feishu 当前 issue 还需要另一个 PR：发布/准备 lingtai-feishu 0.2.1。
  • kernel 还需要另一个 PR：把依赖下限提高到 lingtai-feishu>=0.2.1，防止 fresh install 选择旧包。
